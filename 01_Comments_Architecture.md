# Архитектура комментариев

## Metadata

- **Owner**: Comments Domain Team
- **Status**: Approved
- **Last Reviewed**: 2026-04-29
- **Related ADR**: `ADR-001-materialized-path-comments.md`

## 1. Роль сервиса

Подсистема комментариев отвечает за:

- запись новых комментариев/ответов,
- чтение дерева комментариев с режимами раскрытия,
- выделение ветки по `comment_id`,
- контроль видимости и модерации.

## 2. Ключевой паттерн хранения

Рекомендуемый и уже применяемый подход: **Materialized Path**.

- Каждый комментарий хранит `path` (например `1.4.2`) и `level`.
- Это дает быстрые чтения дерева и веток.
- Для консистентности обязательны транзакции и уникальность на уровне БД.

### Почему выбран Materialized Path

- Быстрый read-path дерева и веток при минимуме сложных join.
- Дешевле в сопровождении, чем nested sets при частых insert.
- Проще реализовать selective expand по ветке.

### Почему не альтернативы

- **Adjacency list only**: простая запись, но дорогое чтение дерева на больших тредах.
- **Nested sets**: хороши для read-only дерева, но сложные и дорогие перестройки при вставке.

## 3. Логические слои

- **Controller/API слой**: валидация входа, выбор режима выборки.
- **Domain service**: расчет `path`, `level`, `number`, бизнес-правила.
- **Repository/ORM**: транзакции, блокировки, индексы.
- **Read model**: сборка дерева в JSON-контракт с `children`.

## 4. Конкурентная запись

При параллельной нагрузке гонки возникают в двух местах:

- `number = max + 1` внутри поста,
- генерация очередного сегмента `path`.

Архитектурные требования:

1. Запись только в транзакции.
2. Блокировка родительского контекста (`FOR UPDATE`).
3. Уникальные ограничения:
   - `(post_id, number)`,
   - опционально `(post_id, path)`.
4. Ретраи при конфликте unique (ограниченное число попыток).

## 5. Чтение дерева

Стратегия чтения:

1. Получить комментарии поста с предрасчетом `short`.
2. Сгруппировать по `replyto_id`.
3. Сформировать корни и рекурсивно присоединить детей.
4. Добавить `children_count`.

Почему это масштабируется:

- плоская выборка + сборка в памяти быстрее, чем N+1 recursive запросы.
- фильтры раскрытия (`expand_all`, `time`, `author_id`) работают как read-strategy без мутаций.

## 6. Риски архитектуры

- глубокие ветки могут давать тяжелый payload,
- широкие обсуждения дают высокую стоимость сериализации,
- смешение режимов фильтрации ломает предсказуемость API.

Дополнительные риски:

- cache invalidation для hot-thread моделей может вести к stale дереву,
- burst комментариев на один пост увеличивает lock contention.

## 7. Архитектурные улучшения до production-уровня

- Read-through cache для популярных тредов.
- Пагинация на уровне корней + lazy-load детей.
- Async-побочные эффекты (уведомления/модерация) строго после commit.
- Отдельные SLA метрики: p95 чтения дерева, p95 создания комментария, collision rate.

## 8. Операционные требования

- Обязательный алерт: рост `unique_conflict_rate` > 1% за 5 минут.
- Обязательный алерт: `comments_read_p95` > 250ms в течение 10 минут.
- Для cache invalidation использовать event `comment.created` и TTL fallback.

## 9. Capacity guardrails

- Максимальная глубина дерева по умолчанию: `<= 15`.
- Максимальный payload списка: ограничивать `root_page_size`.
- При превышении p95 по сериализации включать деградацию: `children` только первого уровня (feature flag).
